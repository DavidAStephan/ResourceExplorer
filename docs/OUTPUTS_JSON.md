# `outputs.json` — programmatic feed

The weekly pipeline publishes a compact JSON summary at
[`/data/outputs.json`](https://davidastephan.github.io/ResourceExplorer/data/outputs.json)
so downstream systems can consume the nowcast without parsing the four
companion CSVs.

This document is the **stability contract** for that endpoint. The
`schema` field at the top of the payload is the version key; consumers
should branch on it.

Generated by [`scripts/build_index.R`](../scripts/build_index.R) on
every weekly Pages deploy.

---

## Current schema: `resourcetracker.outputs.v1`

### Top-level

| field | type | description |
|---|---|---|
| `schema` | string | Schema identifier. Currently `"resourcetracker.outputs.v1"`. Increment when fields are removed or repurposed. |
| `generated_at` | ISO-8601 UTC | Time the JSON was written by the Pages build step. |
| `run_timestamp` | ISO-8601 UTC \| `null` | The `run_timestamp` field from the underlying nowcast (i.e. when `run.R` ran). `null` if no nowcast was produced this run. |
| `horizons` | object | One key per published forecast horizon. See below. |

### `horizons`

Two keys are populated when both horizons ran:

- `current` — the in-quarter (h = 0) nowcast.
- `next_quarter` — the one-quarter-ahead (h = 1) forecast.

Either may be absent when the underlying nowcast didn't produce rows
for that horizon (e.g. very early in a quarter the future forecast may
be skipped).

Each horizon object has the same shape:

| field | type | description |
|---|---|---|
| `quarter_end` | `YYYY-MM-DD` | Last calendar day of the target quarter. |
| `commodities` | object keyed by commodity slug | One entry per commodity nowcast for this horizon. |

### Commodity entries

Keys: `iron_ore`, `coal_met`, `coal_thermal`. (The combined `coal`
bucket was split on 2026-05-20; consumers written before then should
sum `coal_met + coal_thermal` to reproduce the old key.) Future
commodities — if the scope ever extends to LNG or non-Australia — would
appear here as additional keys, never as new top-level fields.

Each commodity object:

| field | type | unit | description |
|---|---|---|---|
| `point_mt` | number | Mt | Point estimate of physical export tonnage for the quarter. |
| `lower_80` | number | Mt | Lower bound of the 80% bootstrap CI. |
| `upper_80` | number | Mt | Upper bound of the 80% bootstrap CI. |
| `lower_95` | number | Mt | Lower bound of the 95% bootstrap CI. |
| `upper_95` | number | Mt | Upper bound of the 95% bootstrap CI. |
| `share_observed` | number | fraction in [0, 1] | Fraction of the target quarter that has been observed in PortWatch at run time. `0` for `next_quarter`. Bands collapse as this grows (see [docs/METHODOLOGY.md § 9](METHODOLOGY.md#9-uncertainty-bands)). |
| `production_spec` | string \| `null` | — | The bridge spec that won the per-commodity backtest and produced the point above. One of `aggregate`, `midas`, `bojo`, `lagged`, `price_aug`, `equal_avg`, `inv_mse`. `null` if no production pick was recorded (e.g. first run after a schema change). |
| `rmse_vs_naive` | number \| `null` | ratio | OOS RMSE of the production spec divided by the seasonal-random-walk RMSE on the 2024+ validation sample. < 1 means the model beats naive. `null` when not available. |

### Example

```json
{
  "schema": "resourcetracker.outputs.v1",
  "generated_at": "2026-05-21T09:30:12Z",
  "run_timestamp": "2026-05-20T11:31:34Z",
  "horizons": {
    "current": {
      "quarter_end": "2026-06-30",
      "commodities": {
        "iron_ore": {
          "point_mt": 236.0,
          "lower_80": 232.3,
          "upper_80": 239.0,
          "lower_95": 232.2,
          "upper_95": 241.7,
          "share_observed": 0.55,
          "production_spec": "bojo",
          "rmse_vs_naive": 0.78
        },
        "coal_met":     { "point_mt": 38.2, "...": "..." },
        "coal_thermal": { "point_mt": 49.6, "...": "..." }
      }
    },
    "next_quarter": {
      "quarter_end": "2026-09-30",
      "commodities": {
        "iron_ore": { "point_mt": 234.7, "share_observed": 0, "...": "..." }
      }
    }
  }
}
```

## Stability contract

What consumers can rely on **until `schema` increments**:

1. **Field presence.** Every field listed above will be present in
   every payload. When the underlying value is missing (e.g. no
   production pick) the field appears as JSON `null`, not omitted.
2. **Field types.** Numeric fields are numbers, not strings.
   `share_observed` is always a fraction in [0, 1], never a percent.
   Mt values are floats in megatonnes, never tonnes.
3. **Additive change is non-breaking.** New top-level fields, new
   horizon keys, and new commodity keys may be added without
   incrementing `schema`. Consumers should ignore unknown keys rather
   than treat them as errors.
4. **Removal or repurposing is breaking.** Renaming or removing a
   documented field, or changing its semantics (e.g. switching `point_mt`
   from Mt to kt), will increment `schema` to `v2`. `v1` will not be
   silently mutated.

What consumers should **not** rely on:

- The set of commodity keys. `coal_met` / `coal_thermal` replaced an
  older `coal` key in 2026-05; future scope changes could similarly
  add or remove commodities without bumping the schema. Iterate
  the `commodities` object rather than hard-coding a list.
- The set of `production_spec` values. The candidate bench grows over
  time (`price_aug` was added in 2026-05, a `demand_aug` is on the
  roadmap). Treat this field as opaque.
- The precise number of decimal places. Use the values; don't parse
  them as strings.

---

## Versioning policy

- `resourcetracker.outputs.v1` — current. Established 2026-05-21 with
  the addition of the Q+1 horizon and the JSON endpoint itself.
- Future versions will be documented here as new sections, with a
  migration note describing what changed.

The four sibling CSVs in `/data/` (`nowcast_current.csv`,
`tonnage_daily.csv`, `tonnage_quarterly.csv`, `bridge_diagnostics.csv`)
are **not** part of this contract; their columns may change without
notice. Consumers needing stability should depend on this JSON.
